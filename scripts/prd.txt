# Overview  
Brokered-Scouting MVP transforms an existing interactive star-map prototype into a minimal, production-ready service that enables EVE Frontier players to create, accept, and fulfill paid scouting runs in under three hours of engineering effort. This utility for data brokers and explorers leverages the EVE Frontier ecosystem and prepares for future integration with PODs (Proof of Discovery).

# Core Features  
1. System Selection
   - Pick source and destination systems via searchable dropdowns with auto-complete
   - Intuitive interface for specifying route endpoints
   - Quick access to frequently scouted systems

2. Dual Route Suggestion
   - Display one fastest route (speed-priority) for efficiency
   - Simultaneously show one safest route (risk-priority) for security
   - Visual differentiation between routes (cyan = speed, teal = safe)

3. Live Risk Overlay
   - Color-code systems by 5-minute aggregated kill-count
   - Green (≤0.3), amber (0.3-0.7), red (≥0.7) risk visualization
   - Regular updates (every 20 seconds) to reflect current danger levels

4. Brokered Scout Job
   - Create job: Define source, destination, reward (ISK), and expiry
   - Accept job: First-come-first-served job claiming mechanism
   - Complete job: Upload route proof with jump list and timestamps
   - Job status tracking (OPEN, CLAIMED, DONE)

5. Route Export
   - Persist jump list and risk metrics as JSON
   - Enable future on-chain verification or POD conversion
   - Standardized format for interoperability with other tools

# Technical Architecture  
## System Components
- Frontend: React with TypeScript, Vite, TailwindCSS, D3.js for star map visualization
- Backend: Node.js with Express for REST API, WebSockets, and business logic
- Data Processing: A* pathfinding algorithm with customizable risk weighting
- Data Sources: EVE Frontier API with in-memory LRU cache (node-cache)
- Storage: SQLite for lightweight persistence of jobs and routes
- Testing: Vitest/Supertest for unit and API testing coverage

## Data Models
- Solar Systems: System IDs, names, coordinates, and connectivity information
- Risk Data: Kill counts aggregated by system, timeframe, and severity
- Routes: Jump sequences with associated metadata (distance, risk metrics)
- Scout Jobs: Structured job records with status, rewards, and completion proofs
- Route Proofs: JSON data structures for verifying scouting completion

## APIs and Integrations
- Routes API: GET /api/routes for pathfinding with risk/speed preferences
- Risk API: GET /api/risk/:systemId for current system risk levels
- Jobs API: CRUD endpoints for job lifecycle management
- EVE Frontier API: Integration for system data and killmail information
- WebSocket Updates: Real-time risk overlay and job status notifications

## Infrastructure Requirements
- Lightweight deployment suitable for containerization (Docker)
- Minimal storage requirements (file-based SQLite)
- CI/CD pipeline through GitHub Actions
- Environment configuration through .env templates
- Responsive design for various screen sizes

# Risks and Mitigations  
## Technical Challenges
- Killmail API latency: Implement caching with 5-second TTL and request debouncing
- Race conditions on job acceptance: Use database UNIQUE constraints for OPEN status
- Large graph performance: Apply progressive loading with quad-tree culling
- Time constraints: Strictly adhere to MVP scope; defer ZK-pods and authentication

## MVP Scope
- Focus exclusively on core brokered scouting functionality
- Limit initial feature set to meet 3-hour development timeframe
- Defer non-essential features (complex authentication, ZK-proofs)

## Resource Constraints
- Optimize for minimal resource utilization (SQLite, in-memory cache)
- Implement efficient rendering for star map visualization
- Use lightweight containerization for deployment
